## Project

A Raycast extension that surfaces [mise](https://mise.jdx.dev/) toolchain information (tools, settings, details) inside Raycast. TypeScript + React, targets macOS and Windows.

## Commands

- `npm run dev` — `ray develop`: start Raycast in dev mode with hot reload.
- `npm run build` — `ray build`: production build.
- `npm run lint` / `npm run fix-lint` — Raycast ESLint config.
- `npm run publish` — publish to the Raycast Store (not npm; `prepublishOnly` is wired to block accidental `npm publish`).

Use `npm` (not `bun`) — the lockfile is `package-lock.json` and Raycast's CLI assumes npm.

## Architecture

- `package.json` is the **source of truth** for the extension surface: every entry in the top-level `commands` array maps 1:1 to a file in `src/<command-name>.tsx` whose default export is the command's React component. Adding a command means editing both.
- `preferences` declared in `package.json` (e.g. `misePath`) are read through `src/preferences.ts`, which casts `getPreferenceValues()` against the ambient `Preferences` types generated by Raycast into `raycast-env.d.ts`. **Do not edit `raycast-env.d.ts` by hand** — it is regenerated by `ray develop` / `ray build` from `package.json`.
- New preferences flow: add to `package.json` → run dev/build to regenerate `raycast-env.d.ts` → consume via `preferences.ts`.
- Current command implementations (`mise-list-all-tools`, `mise-show-settings`, `mise-show-detail`) are **scaffolding only** — they render placeholder `List`/`Detail` items and do not yet shell out to `mise`. Real integration (invoking the `misePath` binary, parsing output) still needs to be built.

## Reference

`raycast-python/` is a separate, unrelated Raycast extension kept in-tree as a reference implementation for patterns (preferences, list/detail commands, metadata). It is not part of this extension's build — ignore it when running `ray` commands from the repo root.

## Conventions

- Prettier: `printWidth: 120`, double quotes (`singleQuote: false`).
- ESLint: extends `@raycast/eslint-config` — don't add custom rules without reason.
- TypeScript `strict` is on; commands use `react-jsx` runtime, so no explicit `React` import is needed.
